# Contributing

欢迎把这个项目当作一个 **Hermes + Spring Boot 学习与实践资料库** 来持续补充。

这个仓库当前更偏向：

- 团队学习材料
- 架构理解整理
- 接入实践示例
- 面向培训和知识沉淀的文档工程

所以贡献建议也优先围绕这些方向展开。

## 1. 推荐贡献方向

你可以优先补充以下内容：

### 文档类

- Hermes 工程化能力分析
- Spring Boot 与 Hermes 的对接实战
- MCP 服务设计经验
- 会话、记忆、上下文治理理解
- 架构图、时序图、对照表
- 团队培训材料和最佳实践

### 示例工程类

- `example-spring-client` 的增强示例
- `example-spring-mcp-server` 的增强示例
- 更贴近真实业务的 MCP 工具示例
- 测试、配置、错误处理示例

### 结构整理类

- 文档导航优化
- 阅读顺序优化
- 团队角色导向的学习索引
- 更适合 GitHub 展示的目录整理

## 2. 贡献前建议先看这些文件

建议先阅读：

1. `README.md`
2. `README-快速开始.md`
3. `docs/README.md`
4. `docs/《Hermes 团队工程化学习手册（拆分版）》.md`
5. `docs/团队工程化学习手册-拆分版/README.md`

如果你是准备补代码示例，建议再看：

- `example-spring-client/README.md`
- `example-spring-mcp-server/README.md`

## 3. 文档贡献原则

为了让这个仓库更适合作为长期学习资料，建议遵守下面这些原则：

1. 优先写清楚“为什么这样设计”，而不只是“怎么做”
2. 优先写清楚“适用场景”和“边界”
3. 尽量保持章节结构清晰，标题明确
4. 如果新增图示，尽量统一放入 `docs/` 体系并使用 Mermaid
5. 如果是对比内容，尽量把维度列清楚
6. 如果是实战内容，尽量给出推荐落地顺序

## 4. 示例代码贡献原则

1. 示例应保持“最小可运行”
2. 示例应尽量便于阅读，而不是堆满复杂框架代码
3. 重要类、方法、配置项建议保留说明性注释
4. 如果新增能力，请同步补 README 或文档导航
5. 如果改动了接入方式，请同步补“适用场景”和“推荐顺序”说明

## 5. 提交内容建议

一个较理想的贡献应尽量包含以下至少一项：

- 文档内容本身
- 目录导航更新
- 示例工程更新
- 对应 README 更新

如果新增了重要主题，建议同时补：

- 一篇完整文档
- 一个在 `docs/README.md` 中可被发现的入口

## 6. 不建议的贡献方式

为了保持这个仓库的学习型定位，暂时不建议：

- 直接把 Hermes 官方完整源码再次复制进来
- 把项目改造成重量级业务仓库
- 把示例工程改得过于复杂，导致失去教学价值
- 在没有文档说明的情况下引入大量新结构

## 7. 最后一句话

这个仓库最重要的目标不是“功能越多越好”，而是：

**让更多团队能更清楚地理解 Hermes、理解它和 Spring Boot 的结合方式，并把这些知识沉淀成可以持续复用的材料。**
