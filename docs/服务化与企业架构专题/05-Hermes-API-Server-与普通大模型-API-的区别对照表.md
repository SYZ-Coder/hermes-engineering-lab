# Hermes API Server 与普通大模型 API 的区别对照表

很多人会觉得：

> Hermes API Server 看起来也像聊天接口，和普通大模型 API 有什么本质区别？

答案是：

**表面像，内核不一样。**

因为普通大模型 API 更像“模型推理接口”，
而 Hermes API Server 更像“Agent 执行接口”。

## 一句话区别

### 普通大模型 API
更像：
**你给它上下文，它给你文本。**

### Hermes API Server
更像：
**你给它任务，它帮你跑一轮 Agent 过程，再把结果给你。**

## 对照表

| 维度 | 普通大模型 API | Hermes API Server |
|---|---|---|
| 本质 | 模型调用接口 | Agent 运行时入口 |
| 处理对象 | prompt / messages | 任务 / 会话 / 工具 / Agent 执行 |
| 会话能力 | 需要调用方自己强维护 | Hermes 自己有正式会话系统 |
| 工具调用 | 你自己额外封装 | Hermes 内部有统一工具系统 |
| 长任务处理 | 你自己扩展 | Hermes 天然更适合任务推进 |
| 上下文治理 | 多数要自己做 | Hermes 内部已有治理能力 |
| 记忆/技能 | 通常要自己搭 | Hermes 已经有相关机制 |
| 落地形态 | AI 能力底层接口 | 更接近智能编排层服务 |

## 为什么这区别很重要？

因为如果你把 Hermes 当普通模型接口来理解，你就会低估它：

- 低估它的会话能力
- 低估它的工具能力
- 低估它的上下文治理
- 低估它和业务系统结合时的价值

## 普通模型 API 更像什么？

更像：
- 一个非常强的“推理引擎”
- 但很多工程能力要你自己补

## Hermes API Server 更像什么？

更像：
- 一个已经具备 Agent 基础设施的“智能执行层”

它不是代替模型，
而是站在模型之上，多做了一层：

- 组织上下文
- 持有会话
- 管理工具
- 处理长任务
- 接外部能力
- 统一执行流程

## 一句话总结

**普通大模型 API 更像能力底座，Hermes API Server 更像企业可接入的智能编排层服务。**
