# Hermes 工具系统为什么设计得比较稳

很多 Agent 系统，一旦开始接工具，问题就会很快暴露出来：

- 工具越来越多，入口越来越乱
- 模型经常调用不存在的工具
- 不同平台可用工具不一致，结果到处出 bug
- 一改工具定义，就把整个会话稳定性搞坏
- 工具调用结果、上下文、权限边界互相缠在一起

Hermes 的工具系统之所以显得“稳”，是因为它在工具越来越多的前提下，仍然把边界和流程收得比较清楚。

### 1. 什么叫“稳”

这里说 Hermes 工具系统稳，不是指“永远不会报错”，而是它在这些方面做得比较扎实：

1. 工具注册路径清晰  
2. 工具暴露面可控  
3. 工具执行链路统一  
4. 工具和主循环职责分离  
5. 工具和上下文、会话、平台之间边界清楚  
6. 出错时不会轻易把整个运行时搞乱

### 2. 不是“到处直接调函数”

Hermes 有一个很清晰的中心化结构：

- 工具实现放在 `tools/*.py`
- 工具注册统一进入 registry
- 工具发现和工具定义生成统一走工具编排层

### 3. 工具“实现”和“编排”是分开的

Hermes 把这些事情拆开了：

- 工具怎么实现
- 工具什么时候可见
- 工具怎么暴露给模型
- 工具怎么执行
- 工具调用失败怎么处理

### 4. 不把所有工具都无脑丢给模型

Hermes 很重视 toolset 的概念。

一个工具要真正进入模型可见面，至少要过几关：

1. 被发现
2. 被注册
3. 属于当前启用的 toolset
4. 通过可用性检查
5. 最终被生成进本轮 tools schema

### 5. 会先判断“能不能用”，再决定“给不给看”

如果依赖条件不满足，Hermes 不会强行暴露给模型。这会把问题拦在“工具暴露前”，而不是等模型调用时再炸。

### 6. 模型只负责“决定调用什么”，不负责“怎么执行”

模型负责判断要不要用工具、给出工具名和参数。真正执行工具的是 Hermes 自己的运行时。

### 7. 工具调用是正式会话的一部分

工具调用不是旁路动作，它是正式对话链路的一部分：

- 模型发起工具调用
- Hermes 执行工具
- 工具结果进入消息历史
- 再把结果回填给模型继续推理

### 8. 工具系统服从上下文治理

工具结果不会无限野蛮生长，而会进入统一治理：

- prompt builder
- context compressor
- prompt caching
- session storage

### 9. 普通工具和子代理分层

- 普通工具调用：主代理自己做
- `delegate_task`：主代理创建子代理去做

### 10. 一句话总结

**Hermes 的工具系统之所以比较稳，不是因为工具少，也不是因为限制多，而是因为它把“工具”当成正式架构层来治理：统一注册、统一暴露、统一分发、统一纳入上下文与会话系统。**
