# Hermes 为什么适合接 Spring Boot

从工程落地角度，更准确的结论是：

**Hermes 和 Spring Boot 在职责分工上天然互补。**

### 1. 不是因为语言兼容，而是因为分工兼容

Hermes 更像：

- 任务理解与能力编排层

Spring Boot 更像：

- 企业业务能力承载层

### 2. 为什么 Spring Boot 很适合当 Hermes 的搭档

因为大多数团队手上的真实系统，本来就已经是 Spring Boot 生态：

- 用户中心
- 订单系统
- 支付系统
- 审批系统
- 报表系统
- 权限系统

### 3. Hermes 不适合吞掉业务系统，而适合“接”业务系统

Hermes 擅长：

- 把自然语言任务翻译成行动
- 选择能力
- 编排工具
- 汇总结果

但它并不适合替代：

- 订单领域逻辑
- 用户规则
- 权限体系
- 审批流程核心规则
- 事务一致性

### 4. 最自然的第一种关系：Spring Boot 调 Hermes

适合走：

- API Server

### 5. 第二种关系：Spring Boot 把事件推给 Hermes

适合走：

- Webhook

### 6. 第三种关系：Spring Boot 把业务能力暴露给 Hermes

适合走：

- MCP

### 7. 为什么 MCP 让两者更像“天然搭档”

- Hermes 负责决定要调用什么能力、何时调用、如何整合结果
- Spring Boot 负责提供稳定、结构化、受控的业务能力

### 8. 为什么特别适合企业内部场景

因为企业内部场景通常：

1. 业务系统已经存在  
2. 系统治理要求高  
3. 智能层更适合后加

### 9. 一句话总结

**Hermes 之所以适合接 Spring Boot，不是因为它们只是技术上能互调，而是因为 Hermes 适合做任务理解与能力编排层，Spring Boot 适合做稳定业务能力层，两者结合后刚好形成一条很自然的企业落地路径。**
