# Hermes 子代理机制为什么设计得比较克制

很多系统的多代理机制一旦放开，很快就会遇到问题：

- 任务拆分越来越失控
- 上下文越来越难管
- 成本越来越不可控
- 调用链越来越深

Hermes 的思路很明确：

**可以委托，但不放飞。**

### 1. 不是把“多代理”当主角，而是把“委托”当工具

Hermes 没有把“多代理系统”做成整个架构的中心，它的子代理更像是：

**主代理在做任务时，临时把一个局部子任务委托出去。**

### 2. 子代理是什么

Hermes 的子代理，就是主代理在运行中临时拉起的一个独立工作单元。

关键词是两个：

- 临时
- 聚焦

### 3. 为什么很多系统的子代理容易失控

因为它们默认接受了这些危险前提：

- 子代理可以无限扩张
- 子代理拿到和主代理几乎一样的权限
- 子代理上下文边界模糊
- 子代理结果没有清晰回收机制

### 4. 子代理限制在“当前任务内部”

在 Hermes 里：

- 主任务还是主代理负责
- 子代理只是辅助完成某个子目标
- 最终对用户负责的，仍然是主代理

### 5. 比普通工具更强，但比系统级多 agent 更收敛

Hermes 的子代理机制卡在一个非常工程化的位置：

**比普通工具更强，但比系统级多 agent 更收敛。**

### 6. 不是“无限自治”，而是“带边界的委托”

子代理通常会受到：

- 任务边界
- 上下文边界
- 工具边界
- 生命周期边界

### 7. 不轻易破坏主会话稳定性

Hermes 的整体设计一直很强调：

**主会话的稳定性优先。**

### 8. 更关注“结果回流”，而不是“代理自治表演”

子代理最重要的事情，是：

**把一个子任务干完，然后把结果有效地带回来。**

### 9. 没有把“多代理”当默认解法

Hermes 的子代理不是默认处方，而更像高级手段。

### 10. 一句话总结

**Hermes 的子代理机制之所以设计得比较克制，是因为它没有把子代理当成长期自治体系，而是把它当成主代理手里一个带边界的任务委托能力：临时创建、聚焦目标、受限运行、结果回流、不过度扩张。**
